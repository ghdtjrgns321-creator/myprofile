# `project_kifrs.html` 문제점과 해결 방향

기존 버전의 전체 톤과 기본 구성은 좋다.  
특히 첫 화면에서 프로젝트 정체성이 바로 보이고, 트러블슈팅과 품질 검증 섹션이 강하다.  
다만 포트폴리오로서의 전달력은 몇 군데에서 손해를 보고 있다.

## 1. 정보 밀도가 높아 첫 독해 부담이 큼

### 문제점
- 첫 화면부터 설명량이 많다.
- 한 카드 안에 핵심 메시지와 세부 근거가 같이 들어 있어 시선이 분산된다.
- 채용자는 보통 1차로 빠르게 훑는데, 현재는 “읽어야 이해되는 페이지”에 가깝다.
- `개요` 섹션에서 핵심 포인트, 문제의식, 비교표가 한 번에 몰려 있다.

### 해결 방향
- 첫 화면에서는 `무엇을 만든 프로젝트인지`, `왜 중요한지`, `무슨 결과를 냈는지`만 먼저 보여준다.
- 세부 설명은 아코디언이나 하위 섹션으로 내려서 2차 독해용으로 분리한다.
- 한 카드에는 하나의 메시지만 남긴다.
- 비교표도 첫 화면에서는 3~4줄 핵심만 남기고, 전체 표는 아래로 내린다.

## 2. 채용자 관점의 핵심 요약이 앞에 없다

### 문제점
- 현재는 프로젝트 설명은 잘 되어 있지만, “그래서 이 사람이 어떤 역량을 보여주는가”가 앞에서 바로 보이지 않는다.
- 읽는 사람이 스스로 해석해야 한다.
- 기술 설명은 강하지만, 채용자용 메시지로 번역된 요약은 약하다.

### 해결 방향
- 헤더 바로 아래에 `이 프로젝트에서 드러나는 역량`을 3개 정도로 명시한다.
- 예시:
  - 제품 감각: 범용 AI와 다른 사용 흐름 설계
  - AI 엔지니어링: Decision Tree와 구조화 출력으로 품질 제어
  - 검증 문화: 620회 테스트로 설계 선택 검증
- 즉, 설명을 “프로젝트 소개”에서 끝내지 말고 “지원자 역량 증명”으로 연결해야 한다.

## 3. 카피 톤이 한국어/영어 혼합이라 완성도가 약간 떨어짐

### 문제점
- 한국어 중심 페이지인데 일부 주요 레이블이 영어다.
- 예:
  - `Back to Portfolio`
  - `Project Overview`
  - `Quality Verification`
  - `GitHub Repository`
- 의도된 영문화 페이지가 아니라면 약간 덜 정돈돼 보일 수 있다.

### 해결 방향
- 페이지 톤을 하나로 통일한다.
- 한국어 중심이면 다음처럼 맞추는 편이 자연스럽다.
  - `포트폴리오로 돌아가기`
  - `프로젝트 개요`
  - `품질 검증`
  - `GitHub 저장소`
- 영어를 유지하고 싶다면 전체를 영어 기준으로 맞추는 편이 낫다.

## 4. `<BR>` 중심 줄바꿈이 많아 문서형 느낌이 강함

### 문제점
- 현재는 문장을 구조적으로 나누기보다 `<BR>`로 시각 줄바꿈을 많이 처리하고 있다.
- 이 방식은 모바일에서 특히 답답해 보일 수 있고, 문서가 HTML 레이아웃이 아니라 메모를 붙여넣은 느낌으로 보일 수 있다.
- 긴 문장이 한 단락 안에 계속 이어져 정보 위계가 약해진다.

### 해결 방향
- 줄바꿈 대신 문단 분리, 리스트 분리, 소제목 분리를 사용한다.
- 하나의 문장 안에 설명이 2개 이상 들어가면 문단을 나눈다.
- 핵심 수치와 결론은 별도 배지, 리스트, 테이블 셀로 분리해서 보여준다.

## 5. 탭과 아코디언의 접근성 의미 부여가 부족함

### 문제점
- 탭 UI는 시각적으로는 잘 작동하지만, 접근성 속성이 부족하다.
- `button`과 `tabpanel`을 쓰고 있지만 `tablist`, `tab`, `aria-selected`, `aria-controls` 연결이 없다.
- 아코디언도 `onclick` 토글만 있고 `aria-expanded` 갱신이 없다.
- 키보드 사용자나 스크린리더 환경에서는 의미 전달이 약하다.

### 해결 방향
- 탭 컨테이너에 `role="tablist"`를 부여한다.
- 각 탭 버튼에 `role="tab"`, `aria-selected`, `aria-controls`를 연결한다.
- 각 패널에는 `aria-labelledby`를 연결한다.
- 아코디언 버튼에는 `aria-expanded`와 `aria-controls`를 추가한다.
- 상태가 바뀔 때 JS에서 해당 속성도 함께 업데이트한다.

## 6. 새 탭 링크 보안 속성이 빠져 있음

### 문제점
- `target="_blank"` 링크에 `rel="noopener noreferrer"`가 없다.
- 큰 문제는 아니지만 기본기 측면에서 빠져 보일 수 있다.

### 해결 방향
- 모든 외부 링크에 아래 형태로 통일한다.

```html
<a href="..." target="_blank" rel="noopener noreferrer">
```

## 7. 첫 화면의 버튼 우선순위가 조금 애매함

### 문제점
- `서비스 바로가기`와 `GitHub Repository`는 있지만, 채용자 입장에서 어떤 순서로 보는 것이 좋은지 안내가 약하다.
- 서비스, 저장소, 상세 설명 중 무엇을 먼저 봐야 하는지 맥락이 없다.

### 해결 방향
- 버튼 우선순위를 분명하게 둔다.
- 예:
  - 1순위: 서비스 보기
  - 2순위: GitHub 보기
  - 3순위: 품질 검증 보기
- 또는 버튼 아래 짧은 설명 한 줄을 넣어 각 링크의 성격을 안내한다.

## 8. 좋은 내용이 많지만 “핵심 성과”가 더 강하게 부각되지 않음

### 문제점
- `1,575건`, `620회`, `301회`, `76%` 같은 수치가 여기저기 흩어져 있다.
- 중요한 숫자인데 첫인상에서 바로 박히지 않는다.
- 그래서 강한 프로젝트인데도 첫 스캔에서는 임팩트가 분산된다.

### 해결 방향
- 헤더 아래 또는 첫 개요 섹션 상단에 핵심 수치 카드 3~4개를 고정 배치한다.
- 예:
  - `1,575건 데이터 적재`
  - `620회 품질 검증`
  - `301회 환각 0건`
  - `검색 적중률 76%`
- 숫자는 본문 중간이 아니라 초반 설득 재료로 써야 한다.

## 9. 좋은 섹션 순서는 유지하되, 개요의 압축이 필요함

### 문제점
- 지금 구조 자체는 나쁘지 않다.
- `개요 → 실제 사용 예시 → 품질 검증 → 트러블슈팅` 순서는 설득력이 있다.
- 다만 `개요`가 너무 많은 역할을 맡고 있어서 뒤 섹션으로 가기 전에 피로가 생긴다.

### 해결 방향
- 전체 섹션 구조는 유지한다.
- 대신 `개요`만 더 가볍게 만들고, 세부 논리는 `품질 검증`과 `트러블슈팅`으로 넘긴다.
- 즉, 페이지 전체를 갈아엎기보다 `개요 섹션 다이어트`가 우선이다.

## 우선순위 제안

### 바로 고치면 체감 큰 항목
1. 첫 화면 핵심 요약 추가
2. 개요 섹션 정보량 축소
3. 카피 언어 통일
4. 핵심 수치 카드 전면 배치

### 그 다음 항목
1. 접근성 속성 정리
2. 외부 링크 `rel` 추가
3. `<BR>` 위주 문장 구조 정리

## 한 줄 결론

기존 페이지는 내용 경쟁력은 충분하다.  
문제는 “무엇이 부족한가”가 아니라, **좋은 내용을 조금 더 빨리 읽히게 만드는 정리 작업이 필요하다**는 점이다.
